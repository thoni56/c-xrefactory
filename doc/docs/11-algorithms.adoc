== Algorithms

The code does not always explain the algorithms that it implements. This chapter tries
to describe essential algorithms used by _c-xrefactory_.

=== Lightweight Include Structure Scanning

Before the server can keep references fresh, it needs to know the project's structure:
which files are compilation units, and how they relate through `#include` directives.
The lightweight scan (`scanProjectForFilesAndIncludes`) discovers this without full
parsing.

==== Why Not Full Parsing?

Full parsing (running the C parser with preprocessor, semantic actions, and reference
creation) is expensive — seconds per file on complex codebases. On a cold start with
thousands of CUs, parsing everything up front would block the user for minutes.

The include structure, however, can be determined by simple text scanning: find lines
matching `#include "..."` and resolve the paths. This is orders of magnitude faster
(e.g. ffmpeg: 2619 CUs scanned in under 1 second vs. minutes for full parsing).

==== The Algorithm

[source]
----
scanProjectForFilesAndIncludes(projectDir, includeDirs):
  │
  ├─ 1. Glob for compilation units
  │     listFilesInDirectory(projectDir, recursive)
  │     Filter to CU extensions (.c, .y)
  │     → discoveredCUs (StringList)
  │
  ├─ 2. For each CU, scan for includes
  │     scanFileForIncludes(cuPath):
  │       Read file line by line
  │       Match #include "filename" (quoted includes only)
  │       For each match:
  │         resolveIncludePath(filename, includerDir, includeDirs)
  │         addIncludeReference(includedFileNum, includerPosition)
  │         Recurse into the included file (transitive)
  │
  └─ 3. Return discoveredCUs for deletion detection
----

==== Include Path Resolution

`resolveIncludePath()` tries directories in order:

1. **Includer's directory** — `#include "foo.h"` in `/project/src/bar.c` looks for
   `/project/src/foo.h` first
2. **Include directories** — walks `options.includeDirs` (from compiler interrogation
   or `.c-xrefrc`)
3. **Fallback** — includer-relative (same as step 1, for robustness)

This matches standard C compiler behavior for quoted includes.

==== What It Produces

The scan populates the same data structures that full parsing would:

`TypeCppInclude` references with `UsageUsed`::
  Created by `addIncludeReference()`. Each reference records which file includes which,
  with the includer's position. These are the edges that Entry Refresh Pass 2 queries
  to find includers of stale headers.

`TypeCppInclude` references with `UsageDefined`::
  Identity references (a file "defines" itself as an includable entity). Created by
  `addFileAsIncludeReference()`.

File table entries::
  New files discovered during include resolution are added to the file table.

The key insight is that the scan produces **the same `TypeCppInclude` references** that
full parsing creates. Code that queries the include graph (like
`collectIncludersOfStaleHeader()` in Entry Refresh) doesn't care whether the references
came from scanning or parsing.

==== Current Limitation: Angle-Bracket Includes

The scan only captures `#include "..."` (quoted includes), not `#include <...>` (angle-bracket
includes). This is deliberate: without the header-filtered sibling optimization (see
Entry Refresh Pass 3 limitations), scanning `<string.h>` and similar ubiquitous system
headers would cause near-full-project sibling explosions.

Full parsing does discover `<...>` includes. On the first request after cold start, Pass 1
reparses the request CU with the full parser, which finds the `<...>` edges. Pass 3 may
then discover additional siblings via those edges — this is the cause of the "double
progress bar" on cold start.

==== Deletion Detection

After scanning, `markMissingFilesAsDeleted()` iterates the file table and marks entries
whose files no longer exist on disk. This handles the case where a user deletes a source
file between sessions. Marked files have their references purged via
`removeReferenceableItemsForFile()` so that navigation doesn't point to deleted files.

Only CUs are checked (they are glob-discoverable). Header files are not checked because
they might legitimately be absent if they're generated or live outside the project tree.

=== Entry Refresh

Entry refresh is the per-request mechanism that ensures the reference table is
up-to-date before any operation executes. It runs at the top of `callServer()`, before
the operation dispatches. See <<06-principles.adoc#,Principles>> for why this is
necessary.

==== Overview

The refresh has three passes, each addressing a different source of staleness.
Pass ordering is critical — later passes depend on data refreshed by earlier ones.

[source]
----
callServer():
  │
  ├─ loadAllOpenedEditorBuffers()
  ├─ closeEditorBuffersNoLongerPreloaded()   ← ghost buffer pruning
  │
  ├─ IF projectContextInitialized:
  │   reparseStalePreloadedFiles(baseArgs)
  │   │
  │   ├─ Pass 1: reparse stale CUs
  │   │   FOR each editor buffer WHERE fileNumberIsStale():
  │   │     IF isCompilationUnit: reparseStaleFile()
  │   │
  │   ├─ Pass 2: reparse includers of stale headers
  │   │   FOR each editor buffer WHERE fileNumberIsStale():
  │   │     IF NOT isCompilationUnit (i.e. header):
  │   │       collectIncludersOfStaleHeader() → list of CU file numbers
  │   │   FOR each collected CU: reparseStaleFile()
  │   │
  │   └─ Pass 3: parse unparsed sibling CUs (cold start)
  │       Find CUs sharing headers with request file but never parsed
  │       FOR each: reparseStaleFile()
  │
  └─ dispatch operation
----

==== Pass 1: Reparse Stale CUs

The simplest case: the user edited and saved a `.c` file, or the client sent a
`-preload` with newer content. The CU itself is stale.

**Detection:** `fileNumberIsStale()` returns true when:

* `lastParsedMtime == 0` — explicitly marked stale (e.g. after refactoring), OR
* The file has a preloaded editor buffer whose `modificationTime` exceeds
  `lastParsedMtime`

**Action:** `reparseStaleFile()` removes all existing references for the file
(`removeReferenceableItemsForFile`), then parses it fresh (`parseFileWithFullInit`).

**Side effect:** Reparsing a CU also refreshes its `TypeCppInclude` references (the
include edges). This is essential for Pass 2.

==== Pass 2: Reparse Includers of Stale Headers

If the user edited a header file, all CUs that include it (transitively) have stale
references — even though those CUs themselves haven't changed.

**Detection:** Same `fileNumberIsStale()` check, but filtered to non-CU files (headers).

**Discovery:** `collectIncludersOfStaleHeader()` walks the reverse include graph:

1. Find all `TypeCppInclude` references where `usage == UsageUsed` and
   `position.file == includerFileNumber` — these are include edges
2. For each includer, if it's a CU, add it to the reparse list
3. If it's another header, recurse (transitive closure)

**Why Pass 1 must come first:** When a user adds a new `#include "foo.h"` to `bar.c`,
both `bar.c` (the includer) and `foo.h` (the includee) may be stale. Pass 1 reparses
`bar.c`, which creates the new `TypeCppInclude` reference for the `bar.c → foo.h` edge.
Pass 2 can then find this edge when looking for includers of `foo.h`. Without Pass 1
first, the edge wouldn't exist yet.

==== Pass 3: Parse Unparsed Sibling CUs

On cold start (no snapshot, or snapshot doesn't cover everything), many CUs have never
been parsed. If the user navigates to a symbol defined in a header, the server needs
references from _other_ CUs that include that header — otherwise navigation only shows
the definition, not all usage sites.

**Discovery:** `parseUnparsedSiblingCUs()` uses a forward-include lookup:

1. Iterate all files in the file table that have `TypeCppInclude` references
2. For each, check if the request file is among its includers
3. If so, collect CUs that also include the same headers ("siblings")
4. Filter to CUs with `lastParsedMtime == 0` (never parsed)

**Scope control:** Only CUs sharing headers with the _request file_ are parsed — not
the entire project. This keeps cold-start latency proportional to the local
neighborhood of the file being worked on.

**Current limitation:** The sibling lookup is based on all shared headers, not just
the header containing the navigated symbol. This can over-parse on projects with
widely-shared utility headers (e.g. `avutil.h` in ffmpeg pulls in ~978 siblings).
A future optimization would filter to sibling CUs sharing the _specific_ header that
defines the symbol being navigated to.

==== Staleness After Refactoring

Refactoring operations (rename, move-function, etc.) modify editor buffers in memory.
The disk files and their mtimes are unchanged, so the normal mtime-based staleness
detection doesn't trigger.

After `serverPerformRefactoring()` completes, `markModifiedEditorBuffersAsStale()`
iterates all editor buffers with `modifiedSinceLastQuasiSave == true` and sets their
file's `lastParsedMtime = 0`. This causes `fileNumberIsStale()` to return true on
the next request, triggering a reparse via Pass 1.

This is called _before_ `quasiSaveModifiedEditorBuffers()`, which clears the
`modifiedSinceLastQuasiSave` flag and updates the buffer's `modificationTime`.

=== Editor Buffers and Incremental Updates

This section describes how `c-xrefactory` handles the reality that source code exists in
two places: on disk as files, and in memory as editor buffers. It also explains the
different update strategies and how references flow through the system.

==== Editor Buffer Abstraction

*_The Duality of Source Code_*

When a user edits code in Emacs, the source code exists in two forms:

* **Disk files**: The saved state on the filesystem
* **Editor buffers**: The current (possibly unsaved) state in the editor

For code analysis to be useful during active editing, c-xrefactory must treat **editor
buffers as the source of truth** when they exist.

*_The Preloading Mechanism_*

When the Emacs client sends a command to the `c-xref` server (like PUSH or NEXT), it uses
the `-preload` option to transmit modified buffers:

----
-preload <original-file> <temp-file>
----

For example:
----
-olcxnext -olcursor=5 /project/foo.c -preload /project/foo.c /tmp/emacs-xxx.tmp
----

**The process:**

1. Emacs creates a temporary file containing the current buffer content
2. The temp file's modification time represents when the buffer was last modified
3. The server loads this into an `EditorBuffer` structure
4. When parsing, the server reads from the temp file (buffer content) instead of the disk file

This ensures that the server analyzes what the user sees in the editor, not the potentially stale disk file.

*_EditorBuffer Lifecycle_*

[source,c]
----
EditorBuffer {
    char *name;                    // Original filename
    char *preLoadedFromFile;       // Path to temp file with buffer content
    time_t modificationTime;       // When buffer was last modified
    bool modified;                 // Content changed (by refactoring)
    bool modifiedSinceLastQuasiSave;  // Dirty flag for quasi-save
    bool preloadedThisRequest;     // Was this buffer preloaded on this request?
    ...
}
----

Editor buffers have different origins and lifetimes:

Preloaded buffers::
  Created by `openEditorBufferFromPreload()` when the client sends `-preload`. These
  have `preLoadedFromFile` set to the temp file path. They persist across requests as
  long as the client keeps sending the preload.

Refactoring buffers::
  Created by editor operations during refactoring (e.g. `findOrCreateAndLoadEditorBufferForFile`).
  These have `preLoadedFromFile == NULL` and `modified == true`. They persist because
  `bufferIsCloseable()` returns false for modified buffers.

Transient buffers::
  Created during normal request processing (parsing). These have `preLoadedFromFile == NULL`
  and `modified == false`. They are destroyed by `closeAllEditorBuffersIfClosable()` at the
  end of each request.

==== Preload Protocol and Ghost Buffer Pruning

The Emacs client sends `-preload` for **all modified buffers** on **every request**. This
is not an optimization — it is a protocol invariant that the server depends on:

* **Presence of `-preload`:** "This buffer is modified in the editor; use this content."
* **Absence of `-preload`:** "This buffer is no longer modified; the disk file is current."

The server uses this to detect when a user saves or closes a file:

[source]
----
server() request loop:
  │
  ├─ clearPreloadedThisRequestFlags()     ← reset all flags to false
  ├─ initServer(pipedOptions)             ← parses options, sets flags on preloaded buffers
  │   └─ openEditorBufferFromPreload()
  │       └─ buffer->preloadedThisRequest = true
  │
  └─ callServer()
      └─ closeEditorBuffersNoLongerPreloaded()
          FOR each editor buffer:
            IF preLoadedFromFile != NULL AND NOT preloadedThisRequest:
              removeReferenceableItemsForFile(buffer->fileNumber)  ← purge stale refs
              deregisterEditorBuffer + freeEditorBuffer             ← close buffer
----

Without ghost buffer pruning, a stale preloaded buffer would persist indefinitely. The
server would keep parsing from the old content even though the user saved the file and
the disk version is newer.

**Reference cleanup is essential:** Simply closing the buffer is not enough. The reference
table still contains entries from the stale parse. `removeReferenceableItemsForFile()` must
be called before closing the buffer, so that the next parse (from the disk file) populates
the table with fresh references without duplicates.

**Navigation sessions are safe:** Sessions store `Reference` structs (position-based), not
`EditorMarker` pointers. Closing a buffer does not create dangling pointers in the
browsing stack.

==== Modification Time Tracking

To implement incremental updates, c-xrefactory tracks when each file/buffer was last parsed.

*_The Fields_*

Each `FileItem` in the file table has:

* `lastParsedMtime` - The modification time when we last parsed this file (any update mode)
* `lastFullUpdateMtime` - The modification time when we last did a FULL update (including header propagation)

These are `time_t` values (seconds since epoch).

*_Dual Semantics_*

The `lastParsedMtime` field has dual semantics depending on context:

* **For disk files**: Stores the file's mtime when it was parsed
* **For editor buffers**: Stores the buffer's mtime (from the preloaded temp file)

This works because `editorFileModificationTime()` abstracts over both:

[source,c]
----
time_t editorFileModificationTime(char *filename) {
    EditorBuffer *buffer = getEditorBuffer(filename);
    if (buffer != NULL && buffer->preLoadedFromFile != NULL) {
        return buffer->modificationTime;  // Buffer time
    } else {
        return fileModificationTime(filename);  // Disk time
    }
}
----

The abstraction is seamless: code can check "has this file changed?" without caring whether it's a disk file or editor buffer.

*_Change Detection_*

To determine if a file/buffer needs reparsing:

[source,c]
----
if (editorFileModificationTime(fileItem->name) != fileItem->lastParsedMtime) {
    // File/buffer has changed since we last parsed it
    reparse(fileItem);
    fileItem->lastParsedMtime = editorFileModificationTime(fileItem->name);
}
----

This pattern appears in:

* `schedulingToUpdate()` - Marks files needing update before batch processing
* `processModifiedFilesForNavigation()` - Detects modified buffers during navigation

==== Update Strategies

`C-xrefactory` has two update strategies that trade off speed against completeness.

*_Fast Update (Default)_*

**When used:**

* Automatic updates before PUSH operations (if enabled)
* Explicit `-fastupdate` command

**What it does:**

1. Checks which source files (.c) have changed (compares modification times)
2. Reparses only those changed source files
3. Updates the references database

**Trade-off:**

* ✅ **Fast**: Only reparses files that actually changed
* ❌ **Incomplete**: Doesn't detect when header files change

**Example:**

----
foo.h modified → fast update → foo.h not reparsed
foo.c unchanged → foo.c not reparsed
Result: foo.c still has stale information about symbols from foo.h
----

*_Full Update_*

**When used:**

* Explicit `-update` command
* When `-exactpositionresolve` is enabled (forces full update)

**What it does:**

1. Checks which files (source OR headers) have changed
2. If a header changed, finds all source files that include it (transitively)
3. Reparses all affected source files
4. Updates the references database

**The algorithm** (`makeIncludeClosureOfFilesToUpdate`):

1. Mark all changed files as `scheduledToUpdate`
2. For each marked file:
   - Find all files that `#include` it (by looking up include references)
   - Mark those files as `scheduledToUpdate` too
3. Repeat until no new files are added (transitive closure)
4. Reparse all marked source files

**Trade-off:**

* ✅ **Complete**: Catches header changes and propagates to all users
* ❌ **Slower**: Can trigger reparsing of many source files if a common header changes

**Example:**

----
common.h modified → full update → finds 50 files that include it
Result: Reparses all 50 source files to pick up header changes
----

*_When Does the Difference Matter?_*

With modern CPUs and SSDs, the performance difference is often negligible for small to
medium projects. The fast update's header-blindness can lead to subtle bugs where
changes don't propagate. Full update is generally safer and more correct.

==== Reference Lifecycle

References flow through multiple stages in c-xrefactory, with different storage
locations and ownership models at each stage.

*_Stage 1: Parsing_*

When a file is parsed:

1. Symbols are discovered (functions, variables, types, etc.)
2. For each symbol, a `ReferenceableItem` is created or looked up in the `referenceableItemTable`
3. Each usage of that symbol creates a `Reference` with a position
4. The reference is added to the referenceable's reference list

**Storage:** `cxMemory` (a custom arena allocator)

**Lifetime:** Lives until the next update that reparses that file

*_Stage 2: The ReferenceableItemTable_*

The canonical storage for all references.

[source,c]
----
ReferenceableItem {
    char *linkName;        // Symbol identifier
    Type type;             // Function, variable, macro, etc.
    Reference *references; // Linked list of all uses
    ...
}
----

**Key properties:**

* Allocated in `cxMemory`
* Persistent across multiple server operations
* Updated incrementally as files are reparsed
* References are **not individually free'd** - they're arena-allocated

*_Stage 3: Session Stacks_*

When a user performs a PUSH operation (browsing a symbol), a session is created:

[source,c]
----
SessionStackEntry {
    BrowsingMenu *menu;      // Selected symbols
    Reference *references;   // COPY of references for navigation
    Reference *current;      // Current position in navigation
    ...
}
----

**Key properties:**

* The `references` list is a **copy** (via `malloc`) of references from the referenceableItemTable
* Each reference is individually allocated with `malloc` (see `addReferenceToList`)
* When the session is destroyed, references are individually `free`'d (see `freeReferences`)
* Sessions are **snapshots in time** - they don't automatically update when the table changes

*_Why Separate Storage?_*

**Memory ownership:**

* Table references: Arena-allocated, freed in bulk
* Session references: Individually allocated, individually freed

If sessions pointed directly to table references, we'd have:

* Dangling pointers when the table is updated
* Double-free errors when sessions are destroyed
* Memory corruption from mixed allocation strategies

**Snapshots vs. live data:**

* The table is the "live" source of truth
* Sessions are working copies for a specific browsing operation
* Users expect their navigation stack to be stable during browsing

*_The Staleness Problem_*

The separation causes a problem: session references can become stale.

**Scenario:**

1. User PUSHes symbol `foo` → Session created with references at lines 10, 50, 100
2. User edits a file, adding lines
3. User navigates with NEXT → Session still points to lines 10, 50, 100 (wrong!)

**Solution:** `processModifiedFilesForNavigation()`

When NEXT/PREVIOUS operations occur, the server:

1. Detects which editor buffers have changed (modification time check)
2. Reparses those buffers (updates referenceableItemTable)
3. **Rebuilds** the current session's reference list from the updated table
4. Preserves the user's navigation position by index

[source,c]
----
// Find user's position in old list
int currentIndex = 0;
Reference *ref = session->references;
while (ref != NULL && ref != session->current) {
    currentIndex++;
    ref = ref->next;
}

// Free old list and rebuild from table
freeReferences(session->references);
session->references = NULL;
for (BrowsingMenu *menu = session->menu; menu != NULL; menu = menu->next) {
    if (menu->selected) {
        ReferenceableItem *updatedItem = lookupInTable(&menu->referenceable);
        addReferencesFromFileToList(updatedItem->references, ANY_FILE,
                                    &session->references);
    }
}

// Restore position by index
ref = session->references;
for (int i = 0; i < currentIndex && ref->next != NULL; i++) {
    ref = ref->next;
}
session->current = ref;
----

This keeps navigation working correctly with live-edited code.

*_Trade-offs of the Incremental Approach_*

**Advantages:**

* Minimal latency - only reparses changed buffers
* Uses editor buffer content (user's current view)
* Preserves navigation position naturally

**Limitations:**

* Like fast update: doesn't reparse includers of changed headers
* Only updates the current session (other sessions on the stack remain stale)
* Only happens during NEXT/PREVIOUS (not other operations)

For typical usage (navigating within files being actively edited), these limitations rarely matter. A fresh PUSH creates a new session with fresh references.

==== Summary

The key insights:

1. **Editor buffers are the source of truth** when they exist (via preloading)
2. **Modification times** are tracked uniformly for files and buffers
3. **Ghost buffer pruning** detects when the client stops preloading a buffer and discards the stale copy
4. **References live in two places**: canonical table (arena memory) and session copies (malloc)
5. **Sessions are snapshots** that can become stale, requiring incremental rebuilding during navigation

NOTE: The Fast Update and Full Update strategies described above are the **legacy**
`-update`/`-fastupdate` path used by the Xref mode and the old `-refactory` mode. In the
current server mode, these are replaced by the <<entry-refresh,Entry Refresh>> algorithm,
which runs automatically on every request and combines the thoroughness of full update
(header propagation via Pass 2) with the speed of fast update (only reparsing what's
actually stale).

=== Server-Side Refactoring

Refactoring was originally implemented as a separate process (`RefactoryMode`): the
editor spawned a fresh `c-xref -refactory` process for each refactoring operation.
This process started cold — it called `callXref()` to rebuild references, performed
the refactoring, and exited.

With the "memory as truth" architecture, this approach doesn't work: the separate
process has no access to the running server's reference table. The solution is to
perform refactoring as a server operation, where the refactoring code runs inside
the already-initialized server with its populated reference table.

NOTE: The legacy `-refactory` path still exists and is exercised by some tests and
the default Emacs client path. It will be removed once all refactorings are validated
in the server path. The `ensureReferencesAreUpdated` calls in the refactoring code are
guarded with `if (refactoringOptions.mode == RefactoryMode)` — they are only needed in
the legacy path, where the server's entry refresh has not run.

==== How Server-Side Refactoring Works

[source]
----
answerEditorAction():
  case OP_REFACTORY:
    serverPerformRefactoring()
      │
      ├─ deepCopyOptionsFromTo(&options, &refactoringOptions)
      ├─ loadAllOpenedEditorBuffers()
      ├─ quasiSaveModifiedEditorBuffers()
      │
      ├─ Create point/mark EditorMarkers from cursor/mark offsets
      │
      ├─ Dispatch on refactoringOptions.theRefactoring:
      │   ├─ AVR_RENAME_SYMBOL    → renameAtPoint(point)
      │   ├─ AVR_EXTRACT_*        → extractFunction/Macro/Variable(point, mark)
      │   ├─ AVR_ADD/DEL/MOVE_PARAMETER → parameterManipulation(point, ...)
      │   ├─ AVR_MOVE_FUNCTION    → moveFunction(point)
      │   ├─ AVR_ORGANIZE_INCLUDES → organizeIncludes(point)
      │   └─ AVR_RENAME_MODULE/INCLUDED_FILE → renameAtInclude(point)
      │
      ├─ markModifiedEditorBuffersAsStale()   ← mark for entry refresh
      └─ quasiSaveModifiedEditorBuffers()     ← update buffer mtimes
----

**Key differences from the legacy `-refactory` path:**

No `callXref()` or `mainTaskEntryInitialisations()`::
  The server is already initialized. The reference table is populated by the entry
  refresh that ran before this request. The refactoring code can directly use the
  references.

No `ensureReferencesAreUpdated()`::
  In the legacy path, this function ran a mini-`callXref()` to populate references
  for the specific symbol being refactored. In the server path, references are already
  fresh from entry refresh. The calls are guarded by `if (refactoringOptions.mode == RefactoryMode)`.

Mark stale on completion::
  After the refactoring, `markModifiedEditorBuffersAsStale()` sets `lastParsedMtime = 0`
  on all files whose editor buffers were modified. This ensures the next request's entry
  refresh reparses them, picking up the post-refactoring state. Without this, the
  reference table would remain stale for files modified by the refactoring (particularly
  files other than the next request's target file).

`parseBufferUsingServer()` calls remain::
  Some refactoring functions call `parseBufferUsingServer()` internally (e.g. to gather
  semantic information for safety checks). In the server path, these calls are harmless
  — they parse into the existing reference table without wiping `cxMemory`.

==== Interactive Dialogs

Some refactorings involve interactive dialogs with the client:

Symbol disambiguation (`beInteractive`)::
  When multiple symbols match the cursor position, `displayResolutionDialog()` sends
  an `<olcx-list>` to the client and calls `beInteractive()`. This works in the server
  path because the server loop is blocked on the call stack — `beInteractive()` reads
  from the pipe via `readOptionsFromPipe()` without conflicting with the outer server
  loop. A local copy of `savedOptions` prevents the inner `callServer()` from clobbering
  the refactoring's option state.

Confirmation (`ppcAskConfirmation` / `waitForUserConfirmation`)::
  The server writes `<ask-confirmation>` followed by `<sync>` and waits for a response.
  The client shows a yes/no dialog and sends `-continue` or `-cancel` back through
  the pipe. The server reads the response directly from stdin (bypassing
  `readOptionsFromPipe` to avoid clobbering the request's option buffer), then continues
  with the operation. Both responses produce output — "yes" may trigger additional work
  (e.g. parsing), while "no" proceeds with what is available. +
  Used by search completeness detection: when unparsed CUs exist, the server asks the
  user whether to parse them all before searching.

Extraction dialog::
  The server computes the extraction and sends `<extraction-dialog>` with the replacement
  strings. The client applies the changes to its buffer. The server's buffer is not
  modified by extraction — the client sends the modified content as a `-preload` on the
  next request.

=== How is an Extract refactoring performed?

The region (mark and point/cursor positions) is sent to the _c-xref_
server in a `-refactory -rfct-extract` command.

The server parses the relevant file and sets some information that can
be used in some prechecks that are then performed, such as structure
check, and then the server answers with

```
<extraction-dialog>
    <str .... /str>
    <str .... /str>
    <str .... /str>
</extraction-dialog>
```

The first string is the code that will replace the extracted code,
such as a call to the extracted function. The second string is the header
part that will preceed the extracted code ("preamble"), and the third is then of
course any code that needs to go after the extracted code ("postamble").

The actual code in the region is never sent to, or returned from, the
server. This is handled completely by the editor extension, and used
verbatim (except if it is a macro that is extracted, in which case
each line is terminated by the backslash) so no changes to that code
can be made.

The pre- and post-ambles might be of varying complexity. E.g. when
extracting a macro, the postamble can be completely empty. When
extracting a function both may contain code to transfer and restore
parameters into local variables to propagate in/out variables as
required.

1. The editor then requests a name from the user that it will use in a
rename operation that renames the default named
function/macro/variable.

*_Two-Phase Architecture_*

Extraction operates in two phases:

1. **Collection Phase (during parsing)**: Parser semantic hooks track control flow by registering synthetic labels and collecting references within the marked region.

2. **Analysis Phase (after parsing)**: The extraction module analyzes the collected control flow data, classifies variables by usage patterns, and generates the refactored code (call site, function definition, postamble).

This separation keeps the parser clean and makes extraction logic independently testable.

=== How does lexem stream management work?

Lexical analysis uses a stack of `LexemStream` structures to handle nested macro expansions. The key insight is that the stream type acts as a **discriminator** for buffer ownership and cleanup strategy.

==== The Stream Types

[source,c]
----
typedef enum {
    NORMAL_STREAM,              // File or local buffer
    MACRO_STREAM,               // Macro expansion
    MACRO_ARGUMENT_STREAM,      // Macro argument expansion
} LexemStreamType;
----

NOTE: Historically, there was also a `CACHED_STREAM` type when the caching mechanism was still active. This confirms that stream types are fundamentally about **buffer ownership and refill strategy** - each type encodes where the buffer came from and how to handle it when exhausted.

**NORMAL_STREAM**::
Buffer from file's `lexemBuffer` or a local temporary. Not allocated from arena memory, so no cleanup needed when stream exhausted.

**MACRO_STREAM**::
Buffer allocated from `macroBodyMemory` arena during macro expansion. Must call `mbmFreeUntil(stream.begin)` when popping from stack to free the arena allocation.

**MACRO_ARGUMENT_STREAM**::
Buffer allocated from `ppmMemory` arena during macro argument expansion. Signals `END_OF_MACRO_ARGUMENT_EXCEPTION` when exhausted (cleanup handled by caller).

==== The Refill Algorithm

When `currentInput` runs out of lexems (`read >= write`), `refillInputIfEmpty()` uses the stream type to decide what to do:

[source,c]
----
while (currentInput.read >= currentInput.write) {
    LexemStreamType inputType = currentInput.streamType;

    if (insideMacro()) {  // Stack not empty
        if (inputType == MACRO_ARGUMENT_STREAM) {
            return END_OF_MACRO_ARGUMENT_EXCEPTION;
        }
        // Only free MACRO_STREAM buffers (allocated from macroBodyMemory)
        if (inputType == MACRO_STREAM) {
            mbmFreeUntil(currentInput.begin);
        }
        currentInput = macroInputStack[--macroStackIndex];  // Pop
    }
    else if (inputType == NORMAL_STREAM) {
        // Refill from file
        buildLexemFromCharacters(&currentFile.characterBuffer, ...);
    }
}
----

==== Key Invariant

**Stream type must match buffer allocation:**

* `MACRO_STREAM` → buffer allocated from `macroBodyMemory`
* `NORMAL_STREAM` → buffer NOT from macro arenas
* `MACRO_ARGUMENT_STREAM` → buffer from `ppmMemory`

Violating this invariant causes fatal errors when trying to free buffers from the wrong arena.

==== Common Bug Pattern

Pushing a `NORMAL_STREAM` onto the macro stack, then trying to free it as if it were `MACRO_STREAM`:

[source,c]
----
// WRONG: Blindly freeing without checking type
mbmFreeUntil(currentInput.begin);  // Fails if currentInput is NORMAL_STREAM!

// CORRECT: Check type first
if (inputType == MACRO_STREAM) {
    mbmFreeUntil(currentInput.begin);
}
----
