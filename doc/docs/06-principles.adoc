== Principles

=== Reference Database and Parsing

The persisted snapshot (`.cx` files) stores all file-scope and wider symbols — including
`static` functions and variables. Only block-local symbols (variables inside function
bodies, labels) are excluded. The visibility boundary is `VisibilityLocal` in
`getSymbolCxrefProperties()`: block-locals get `VisibilityLocal`, everything else
(including file-scope statics) gets `VisibilityGlobal`.

Block-local symbols are reconstructed by reparsing the file of interest.

NOTE: The in-memory reference table holds all symbols encountered during parsing,
including block-local ones. The snapshot excludes only block-locals because they
can always be reconstructed by reparsing.

=== Memory As Truth

After startup, the in-memory `referenceableItemTable` is authoritative. The `.cx`
files are a snapshot for surviving process restarts, not a tiebreaker. No operation
reads from `.cx` after `loadSnapshotFromStore()` returns.

The server is long-lived and the only writer of its `.cx` files. A disk read
post-startup means either re-reading our own output or preferring a stale snapshot
over fresh in-memory state — neither is correct. Earlier in the project several
disk-as-truth code paths silently produced _some_ answer, just not the right one;
those bugs were hard to find precisely because the disk read succeeded.

==== Enforcement: the disk-read tripwire

`scanCxFile()` is the only entry point that opens a `.cx` file for reading. It
asserts that `snapshotLoadComplete` (a static flag in `cxfile.c` set at the end of
`loadSnapshotFromStore()`) is still false. Any later call aborts with a message
pointing here.

Writes — periodic save, save on exit — are not affected; the tripwire is on the
read path only. A future feature that requires post-startup disk access means this
principle has to be revisited.

=== Editor Buffers Are the Content Source

In server mode the parser never reads a source file from disk more than once.
Every file the server parses is backed by an in-memory `EditorBuffer`. Its text
is loaded from disk exactly once — lazily, the first time the file is parsed
(`findOrCreateAndLoadEditorBufferForFile` creates the buffer and calls
`loadFileIntoEditorBuffer`). After that the buffer's `textLoaded` flag is set and
the parser always reads the buffer; the disk file is never consulted again.
"Reparsing" a file re-parses its _buffer_ content, not the current contents of the
file on disk.

The only thing that refreshes an existing buffer is the client. When the editor
sends `-preload <file> <tmpfile>`, `openEditorBufferFromPreload` compares the temp
file's modification time against the buffer's and reloads the buffer from the temp
file if they differ. There is no disk polling and no background filesystem sync:
**the editor is the source of truth for file content, and the server mirrors it
through preloads.**

Consequences:

Editor-open files stay current::
  Files the user is editing are sent as preloads, so their buffers track the
  editor in real time. This is the case the design optimizes for.

Saved or closed files re-sync from disk::
  When the user saves or closes a file the editor stops preloading it; ghost-buffer
  cleanup (`closeEditorBuffersNoLongerPreloaded`) frees the now-stale buffer and
  re-reads the file from disk, so the saved content is picked up.

Behind-the-back changes are not detected (known limitation)::
  A file the server has buffered but the editor does not have open — a header
  reached through `#include`, or a file visited by an earlier browse — is never
  re-preloaded, so a change to it _on disk alone_ is invisible. This bites real
  workflows: `git switch`, regenerated files (`*.tab.[ch]`, `lexem.c`), scripts
  that write sources. It is a limitation of the current model, not a deliberate
  exclusion; the likely fix is a per-request disk-mtime recheck for non-preloaded
  buffers (extending the re-sync above to fire on mtime, not only on un-preloading).
  Deferred.

NOTE: A test that changes a file's content on disk between requests _without_
sending a matching `-preload` is exercising this behind-the-back case, not a real
editing scenario. To simulate an edit, send the new content as a preload.

=== Continuously Fresh References

The reference table must be kept fresh on every request. This is the central design
challenge in a request/response architecture where:

* The server is long-lived, holding state across many requests
* Source files change between requests (user edits, saves, closes buffers)
* Each operation (navigation, refactoring, completion) depends on accurate references
* The server cannot know _when_ changes happen — it only discovers them at request time

The principle is: **before each operation, detect the staleness the editor reports
and the structural changes the scan discovers.** Changes the editor never reports —
a buffered file altered behind its back — are a known blind spot (see _Editor Buffers
Are the Content Source_).
There is no single mechanism for this; instead, multiple complementary strategies
cooperate:

Editor edits and saves::
  Unsaved changes arrive as preloads; on save the editor stops preloading and
  ghost-buffer cleanup re-reads the file from disk. Detection is editor-driven — the
  preload and un-preload transitions — not a disk poll.

Editor buffer divergence::
  The user's editor may hold unsaved changes. The client transmits these via
  `-preload`, and their modification time is tracked separately from the disk file.

Editor buffer disappearance::
  The user saves or closes a buffer. The client stops sending `-preload` for it.
  The server must detect this absence and discard the stale in-memory copy ("ghost
  buffer pruning"), falling back to the disk file.

New files and deleted files::
  Files may be added to or removed from the project between requests. The lightweight
  scan discovers new compilation units; `markMissingFilesAsDeleted()` handles removals.

Refactoring side effects::
  A refactoring modifies editor buffers in memory, but the disk files and mtimes
  are unchanged. Modified buffers must be explicitly marked stale so that subsequent
  operations see fresh references.

Include graph changes::
  A stale header affects all compilation units that include it, transitively. Reparsing
  a stale CU refreshes its include references, which then allows discovering other CUs
  affected by the same header change.

Each of these is handled by a specific mechanism in the entry refresh algorithm
(see <<11-algorithms.adoc#entry-refresh,Entry Refresh>> in the Algorithms chapter).
The key insight is that all of them must run _before_ the operation dispatches — there is
no lazy or deferred refresh. The cost is paid up front so that every operation sees a
consistent view of the codebase.

=== Minimal Reparse Set

Full freshness does not require full reparsing. Parsing is the most expensive operation
the server performs, so the system must determine the **minimal set of compilation units**
that need reparsing to make the reference table consistent.

The strategies, in order of increasing scope:

Only stale files::
  If a `.c` file changed, only that file needs reparsing. No other CU is affected
  unless they share headers.

Include-graph-directed propagation::
  If a header changed, only CUs that include it (transitively) need reparsing — not
  all CUs in the project. The include structure (`TypeCppInclude` references) provides
  the reverse lookup.

Sibling CU discovery::
  On cold start, not all CUs have been parsed. Rather than parsing the entire project
  up front (which can take tens of seconds on large codebases), the server discovers
  which unparsed CUs share headers with the request file and parses only those. CUs
  unrelated to the current request remain unparsed until needed.

Skip already-fresh files::
  Mtime comparison avoids reparsing files that haven't changed since their last parse,
  even if they were discovered as potential candidates.

Reparsing only a subset is safe because **an edit clears the edited file's
references**. When a file arrives as a preload, `removeReferenceableItemsForFile`
strips _that file's_ references, whatever its type. Re-creating them is a separate
step whose target depends on the file: a compilation unit is reparsed directly; a
header is only stripped, and its including CUs (found through the reverse-include
graph) are reparsed to re-emit its declarations. In both ordinary cases the edit
invalidates and the reparse rebuilds: an edited compilation unit strips and rebuilds
its own references; an edited header is stripped and its including CUs reparse to
re-emit them.

There is one case the strip does **not** cover. Which declarations a compilation unit
emits into an included header can depend on the CU's _own_ preprocessor state, so a
header-position reference can be left stale by an includer's edit alone — even though
the header itself was never touched.

For example, a header declares a function under `#ifdef FEATURE`. A CU that defines
`FEATURE` and includes the header records that declaration at the header's position.
If the CU later removes the `#define` and is reparsed, `removeReferenceableItemsForFile`
strips only the CU's _own_ positions — not the header position it emitted into. The
header was never edited, so nothing strips the orphaned declaration, and the reparse
gates it out rather than re-emitting it. The reference lingers.

Closing this gap requires reparsing by include structure — stripping and rebuilding the
affected header's contributions — gated so the cost is paid only when an edit actually
changes what a CU emits, not on every edit.

The goal is **interactive-speed response times**: the user should not wait for the
entire project to reparse when they changed one file. On a large codebase (e.g. ffmpeg
with 2600+ CUs), the difference between "reparse everything" and "reparse the 7 CUs
that share the edited header" is the difference between minutes and milliseconds.

=== Preserve Parsed Work

Parsing is the most expensive operation the server performs. On large codebases, a full
parse can take tens of minutes. Some operations — like a project-wide search on cold
start — may require parsing thousands of CUs. The user may abort (Ctrl-g) or the
server may crash during such long-running operations.

The principle is: **parsed work is valuable — save it incrementally so that progress
survives interruption.**

During long-running parse operations, the server periodically saves the reference table
to the persisted snapshot. If the operation is interrupted, the next startup restores
the snapshot and only the remaining unparsed CUs need processing.

The snapshot save is safe at any point because the `staleify` mechanism
(`markPreloadedFilesAsAncient`) ensures that editor buffer state is not persisted — only
disk-file-derived references are saved, and mtime-based validation remains correct at
next startup.

**Measured cost:** On ffmpeg (2600+ CUs), a full snapshot save takes approximately
1 second. This is acceptable at intervals (e.g. every 100 CUs) within a long-running
parse loop — a ~1% overhead that prevents losing minutes of work on interruption.

=== Project Scope

A **project** is defined by a `.c-xrefrc` configuration file. The directory containing
this file is the **project root**. On auto-detection, the server walks upward from the
request file to locate the nearest `.c-xrefrc`.

Source coverage::
  All compilation units under the project root are discovered by the lightweight scan.
  Headers are discovered transitively through `#include` directives in those compilation
  units. Additional source directories can be listed in the configuration file to include
  files outside the root (e.g. shared libraries, multi-repo layouts).

Include paths::
  The `-I` option in the configuration file adds directories that the preprocessor
  searches for `#include` directives — the same semantics as the compiler flag. Files
  resolved through include paths are parsed as part of the project and available for
  navigation.

Project identity during navigation::
  When the user navigates to a file outside the project root, the server must determine
  whether the file belongs to the current project. A file is considered in-scope if it
  resides under the project root, an additional source directory, or a configured include
  path. Without this, navigating into an included header outside the project tree would
  be misinterpreted as leaving the project.

NOTE: The include-path check is broad: _any_ file under an `-I` directory is accepted,
not only files that were actually included. A compilation unit that happens to reside
next to included headers will be treated as in-scope even if the project never references
it. This is a known simplification; a precise check would require distinguishing
"encountered during parsing" from "registered by a transient request," which the current
file table does not support.

=== Terminology

The following terms are used throughout the documentation and codebase. Preferred terms
are listed alphabetically; terms to avoid are noted to reduce ambiguity.

Cold start::
  Server startup with no persisted snapshot available. All compilation units must be
  fully parsed to populate the reference table. Same code path as warm start, just more
  work.

Compilation unit (CU)::
  A source file that is directly compiled (`.c`, `.y`). Discovered by globbing the
  project directory. Distinguished from header files, which are included transitively
  and not compiled independently.

Include structure::
  The graph of `#include` relationships between files, represented in the reference table
  as `TypeCppInclude` references. Cheap to build (text scanning for `#include` lines).
  Separate from symbol references, which require full parsing.

Initialization::
  The first-request setup: discover project, load options, interrogate compiler, restore
  snapshot, scan project structure. Happens once per session. Not the same as cold start
  — initialization happens on every session, cold start describes the _absence_ of a
  snapshot.

Lightweight scan::
  Discovery of project structure by globbing for compilation units and text-scanning
  `#include` lines, without full parsing. Populates the file table and include structure.
  Replaces `-create` and the `callXref()` pre-refactoring pattern (ADR 22).

Persisted snapshot::
  The `.cx` files on disk. A point-in-time copy of the reference table from a previous
  session, loaded at startup to avoid a full parse. May be stale — reconciled against
  the filesystem via mtime comparison. +
  _Avoid "disk db", "cache", "reference database"_ when referring to the `.cx` files.
  A snapshot is not queryable (no search operations on disk), has no invalidation
  semantics (unlike a cache), and is not the source of truth (unlike a database).

Preload::
  Editor-buffer content the client sends for files the user has modified but not yet
  saved, via `-preload <filename> <tmpfile>` arguments. The server reads the temp file
  into an `EditorBuffer` with its modification time. Preloaded buffers persist across
  requests; on each preload the server compares the new temp file's mtime against the
  existing buffer's to decide whether to reload content. Entry refresh then reparses
  any buffer whose mtime has advanced past `lastParsedMtime`. Unmodified files are not
  preloaded — the server reads disk directly.

Project::
  A set of source files governed by a single `.c-xrefrc` configuration file. Defines
  which files are parsed, which include paths are searched, and where the persisted
  snapshot is stored. Identified by the section name in the configuration file.

Project root::
  The directory containing the `.c-xrefrc` configuration file. All compilation units
  under this directory are discovered by the lightweight scan; headers are discovered
  transitively through `#include` directives. On auto-detection, the project root is
  found by walking upward from the request file.

Reference table::
  The authoritative in-memory state during a running session. Comprises the
  `referenceableItemTable` (symbol references), the file table (file entries with
  modification tracking), and `TypeCppInclude` references (include structure). Populated
  by parsing, snapshot restoration, and lightweight scanning. +
  _Avoid "in-memory db", "memory db"._ It is the live working set, not a database in
  the traditional sense.

Restoring::
  Loading a persisted snapshot into the reference table at startup. After restoration,
  mtime comparison determines which entries are fresh and which need reparsing. +
  _Avoid "loading the database"._

Saving / persisting::
  Writing the current reference table state to `.cx` files. Only disk-file-derived
  references are persisted; references from unsaved editor buffers are excluded so that
  mtime-based validation remains correct at next startup. +
  _Avoid "writing the database"._

Staleness::
  A file is stale when its content has changed since it was last parsed. Detected by
  comparing `lastParsedMtime` against the current file modification time (from disk or
  editor buffer). Stale files trigger reparsing during the entry refresh (ADR 20).

Steady state::
  The server is initialized and processing requests. Staleness detection and incremental
  reparsing happen per-request. The reference table is authoritative. +
  _Avoid "hot start"_ — ambiguous.

Symbol references::
  The detailed reference information (positions, usage types) for identifiers, created by
  full parsing. Expensive to produce. Distinguished from include structure, which is cheap.

Entry refresh::
  The per-request mechanism that ensures the reference table is up-to-date before executing
  an operation. Covers both _staleness_ (content was known but is now outdated) and
  _unknown content_ (file discovered by scan but never parsed). Uses the include structure
  to determine what needs reparsing. Runs on every request in `callServer()`. +
  See Server Mode Flow in <<09-code.adoc#,Code>> for implementation details.

Warm start::
  Server startup with an existing persisted snapshot. Most compilation units are fresh
  (snapshot mtime matches disk mtime); only stale ones need reparsing. Same code path as
  cold start, less work.
