== Functional Overview

`c-xrefactory` helps a programmer navigate, understand, and safely improve C and Yacc
source from within the editor. This chapter catalogues that functionality and sketches how
the server serves a request; the project's origins are in <<history,History>> at the end.

=== Main functionality

A programmer constantly needs to navigate, understand and improve the
source code in order to lessen the cognitive load for understanding
and making changes.

`c-xrefactory` provides two sets of functions for this directly from
within the editor

- navigation, searching and browsing symbols
- automated refactorings, i.e. non-behaviour changing edits

C and Yacc source code is supported.

==== Navigation

A user can navigate all references of a symbol, limited to the
semantic scope of that symbol, by "Goto Definition" and then navigate
using "Next/previous Reference". This is a fast way to inspect where a
symbol is used.

NOTE: This also applies to non-terminals and semantic attributes in
Yacc grammars!

==== Browsing

It is also possible to search for definitions or symbols containing a
string, including possible wildcards. This will present various forms
of "menues" where filters can be applied and a found symbol be
inspected.

==== Completion

As `c-xrefactory` have information about symbols and their semantic scope,
it can also provide semantically informed completions and suggestions.

==== Automated refactorings

In his book "Refactoring" Martin Fowler describes a large number of refactorings,
changes to source code that does not change the behaviour, but improves it structure and
readability. For each refactoring they describe step by step which edits to make to
apply the refactoring manually.

The natural next step was of course to attempt to automate this in editors or IDEs,
which started to happen.

In an article from 2001 Martin pronounces `Xref`, the ancestor to `c-xrefactory`, to be
the first tool to cross "Refactorings Rubicon", being able to extract a function
semantically correct.

The term "automated" means that some software can examine the source
code and quickly and safely modify it using patterns from the list of
possible refactorings, without user interaction. Many refactorings in
the book, and on the website, are applicable mostly for OO-languages,
but many also apply to C. `c-xrefactory` can perform some of
them. More are considered for implementation.

- "Rename Symbol" - change the name of a variable, type, function only for the semantic scope of the symbol
- "Extract Macro/Function" - a region of the code can be extracted to a new function or macro
- "Organize Includes" - clean up a list of #include directives by particioning and sorting them
- "Rename Included File" - rename the file in the #include directive and update all other #include directives of that file
- "Move Function To Other File" - move a function to another file, automatically add an extern declaration in an appropriate header file and ensure that is included in the file where the function originally was

Using these automated refactorings it is much easier and safer to
continuously maintain and improve the quality of any code base.

=== Configuration

A project is configured by a `.c-xrefrc` file in the project root, found by upward search
from the file being worked on; it records the source roots, include paths, and defines.
(The older user-central `~/.c-xrefrc` with per-project sections still works.) See
<<06-principles.adoc#,Principles>> (Project Scope) for how project boundaries are
determined.

=== Operational model at a glance

`c-xrefactory` runs as a long-lived server that the editor drives one request at a time.
Per request:

1. The editor sends the operation (navigation, search, completion, or refactoring) along
   with the contents of any buffers it has open, as preloads.
2. Before anything else, the server *refreshes* the in-memory reference table — reparsing
   whatever changed since the last request, so the table reflects current reality.
3. The operation runs against that table, and the server replies — positions to jump to, a
   menu, or edits to apply.

Two design points to carry forward:

*The table is refreshed whole, before every operation.* Each operation simply trusts that
the references it reads are current; it never decides what to refresh. (One could argue an
extract, say, strictly needs only the edited unit's references fresh — but that is a latent
optimization, not how the system is wired: the refresh layer owns correctness, and
operations stay ignorant of it.) The mechanism is
<<08-algorithms.adoc#entry-refresh,Entry Refresh>>; the principle is in
<<06-principles.adoc#,Principles>>.

*Parsing serves two masters.* The parser both populates the reference table and collects
information specific to the current operation — the symbol under the cursor, the region to
extract, the context for a completion. Keeping those roles distinct matters: an
operation-specific parse must still leave the table correct.

=== LSP

The LSP protocol is a common protocol for language servers such as
`clangd` and `c-xrefactory`. It allows an editor (client) to interface
to a server to request information, such as reference positions, and
operations, such as refactorings, without knowing exactly which server
it talks to.

Recent versions of `c-xrefactory` have an initial implementation of a
very small portion of the LSP protocol. The plan is to fully integrate
the functionality of `c-xrefactory` into the LSP protocol. This will
allow use of `c-xrefactory` from not only Emacs but also Visual Studio
Code or any other editor that supports the LSP protocol.

==== LSP Protocol Limitations

The LSP protocol was designed for single-shot, non-interactive operations.
This creates constraints for `c-xrefactory`'s advanced refactorings:

**Interactive Refactorings**: `c-xrefactory`'s extract/parameter operations
require multi-step user input (names, positions, declarations). LSP's
`textDocument/codeAction` doesn't support interactive dialogs.

**Symbol Browsing**: `c-xrefactory` provides interactive symbol browsers
with filtering and keyboard navigation. LSP returns flat reference lists
with no standard for interactive UI.

**Strategy**: The LSP implementation aims to:

- Provide basic IDE features (definition, completion, simple refactorings)
  to modern editors
- Expose `c-xrefactory`'s advanced refactoring capabilities where possible
- Keep the Emacs client as the primary interface for full interactive features

LSP serves to make `c-xrefactory` more accessible while the Emacs client
probably will remain the gateway to its complete refactoring power.

=== History

The _c-xref_ program was, historically, a mish-mash of features in one program — a major
cause of the source-level mess being untangled (see <<10-roadmap.adoc#,Roadmap>>). It was:

- a generator for persistent cross-reference data
- a reference server for editors (cross-reference, navigation, completion)
- a refactoring server — the first tool to cross the "Refactoring Rubicon"
- [.line-through]#an HTML cross-reference generator# (removed)
- [.line-through]#a C macro generator for structure-fill and similar functions# (removed)

The first three are the unique value of the project. The HTML generator — likely the
original purpose, the root of the ancestor `Xrefactory` — was superseded by tools like
Doxygen and removed; the macro generator was a hack that obstructed modern building and
refactoring and was removed. The other capabilities were originally bolted on without
re-architecting the C sources, which is the tail this project is still grooming.
