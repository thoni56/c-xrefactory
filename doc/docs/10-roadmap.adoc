== Roadmap

This chapter records where c-xrefactory is going — the architectural direction and the
feature vision. The mechanisms behind the "done" items live in their home chapters
(<<06-principles.adoc#,Principles>>, <<08-algorithms.adoc#,Algorithms>>); implementation
detail is in <<17-major-codebase-improvements.adoc#,Major Codebase Improvements>>.

=== Guiding Principles

* **Incremental improvement**: Each step should provide immediate value while moving toward long-term goals
* **Test-driven modernization**: Maintain 85%+ test coverage to enable confident refactoring
* **Backward compatibility**: Preserve existing Emacs workflows while enabling modern IDE integration
* **Architectural simplification**: Replace artificial mode distinctions with unified, smart on-demand behavior
* **Legacy code respect**: Work with the existing 1990s codebase thoughtfully, not against it

=== Convergence: Memory as Truth

In-memory references are the single source of truth — every operation (navigation,
search, completion, refactoring, unused detection) reads the in-memory
`referenceableItemTable`, and the `.cx` files are a disposable startup snapshot. This
convergence is **complete**; the mechanism lives in <<06-principles.adoc#,Principles>>
(Memory As Truth, Continuously Fresh References) and <<08-algorithms.adoc#,Algorithms>>
(Entry Refresh, Lightweight Scanning). What follows is status and what remains.

==== Done

* *Separate sync from dispatch* — entry refresh reparses stale files before every
  operation (link:/workspace/1/decisions/C-xrefactory#20[ADR-0020]).
* *Lightweight file-structure scanning* — CU and include discovery without full parsing
  (link:/workspace/1/decisions/C-xrefactory#22[ADR-0022]).
* *Refactoring as a server operation* — all refactorings run via `OP_REFACTORY`;
  `RefactoryMode` and the `-refactory` process removed.
* *Search from the in-memory table* — `searchInMemoryTable()` replaced the disk scan;
  the last disk-as-truth read is gone.
* *Disk snapshot* — `.cx` loaded and mtime-validated at startup, saved on exit and
  periodically; no mandatory `-create`.

=== Optimization

Once the core architectural path (lightweight scan, entry refresh, memory-as-truth) is functional, targeted optimizations can reduce cold-start latency and improve responsiveness on large projects.

[cols="1,3"]
|===
|Optimization |Description

|**Header-filtered sibling parsing**
|Entry refresh pass 3 currently parses all CUs sharing _any_ header with the request file. On large projects this pulls in a large fraction of all CUs (e.g. on ffmpeg, navigating `AudioFIRContext` from `af_afir.c` triggers 2481 sibling CUs because of widely-shared utility headers). When the operation is navigation, only CUs sharing the header that _defines the navigated symbol_ are relevant. Filtering to that header reduces the scope dramatically: `AudioFIRContext` is defined in `af_afir.h`, which only 3 CUs include — so 2481 → 2 sibling CUs to parse. Challenge: pass 3 runs during sync (before dispatch), but the symbol is only known during the operation. The solution must bridge this gap — e.g. by triggering targeted sibling parsing from the operation after the symbol's defining header is identified. Measured cold-start PUSH on ffmpeg `af_afir.c`: scan 2.7s, pass 3 round 1 19s, pass 3 round 2 19s (from `<...>` includes discovered by full parse), total 42s. Parsing dominates — the scan is not the bottleneck. Worst case: core API headers like `avcodec.h` (614 CUs) or `internal.h` (1171 CUs) — parallel parsing may be needed for these.

|**Extend scan to capture `<...>` includes**
|The lightweight scan currently only matches `#include "..."`. Projects like ffmpeg use angle-bracket includes for internal headers (with `-I` paths), so the scan misses roughly half the include structure. Extending to `<...>` requires header-filtered sibling parsing first — without that filter, ubiquitous system headers (`string.h`, `stdlib.h`) would cause pass 3 to pull in nearly every CU in the project. Also needs care to avoid recursively scanning system headers outside the project tree.

|**Lexing cache re-introduction**
|The lexing cache (pre-tokenized file content) was present in the original codebase but lost during restructuring. Re-enabling it avoids repeated lexing of the same file across multiple parse passes.

|**Parallel parsing**
|Even with header-filtered sibling parsing, core API headers in large projects can fan out to hundreds of CUs (e.g. `avcodec.h` in ffmpeg is directly included by 614 CUs, `internal.h` by 1171). Parallel parsing of independent CUs could reduce wall-clock time roughly linearly with core count. The main challenges are: the global mutable state throughout the parser and symbol table (would require per-thread working sets with merge), CX memory allocation (currently a single arena), and the file table (shared, needs synchronization or partitioning). This is a significant engineering effort but may be the only way to achieve acceptable latency for navigation targets in widely-included headers.
|===

Consequences of the current entry refresh implementation, each deferred because it has low
practical impact or dissolves with planned work. Mechanics are in
<<08-algorithms.adoc#entry-refresh,Algorithms>> (Entry Refresh).

|Clearing a CU's outgoing include refs before re-scan

|Cold-start double progress bar — scan finds `"..."` siblings, the later full parse finds
 `<...>` siblings
|Extending the scan to `<...>` includes (see Optimization)

|Index-based sessions (see Convergence → Remaining)

=== Feature Vision: LSP Support (explorative)

The last and least-defined direction: expose c-xrefactory through the Language Server
Protocol so VS Code, Neovim, and other LSP editors can use its navigation and — as far as
the protocol allows — its refactorings. How far that reaches is genuinely open. Basic
features (definition, references, completion) map cleanly, but c-xrefactory's interactive
refactorings and symbol browsers have no natural LSP equivalent, so the Emacs client will
likely remain the gateway to its full refactoring power. Surfacing even some refactorings
in an LSP environment is the prize worth chasing; the reach is unknown until we get there.

See <<03-functional-overview.adoc#,Functional Overview>> for the current LSP state and
protocol limits, and <<11-planned-features.adoc#,Planned Features>> for the implementation
steps.

The one durable design point: *LSP document events are entry-refresh sync points.*
`didOpen` / `didChange` / `didSave` correspond exactly to the native path's preload and
un-preload staleness signals, and `definition` / `references` are operations over a table
those events have already made current. So the LSP path needs no architecture of its own —
it reuses entry refresh (<<08-algorithms.adoc#,Algorithms>>), triggered by different events.

=== Further refactorings

c-xrefactory crossed the "Refactoring Rubicon" early, but only a fraction of the
refactorings that apply to C are automated today. Extending the catalogue — both deeper
versions of what exists (move-function follow-ups, delete-function) and new operations —
is a standing source of value. Several candidates are specified in
<<11-planned-features.adoc#,Planned Features>>.
